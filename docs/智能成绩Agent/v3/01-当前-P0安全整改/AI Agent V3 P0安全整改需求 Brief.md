# AI Agent V3 P0 安全整改需求 Brief

> 角色：产品部  
> 日期：2026-07-13  
> 状态：待研发部技术方案评审  
> 优先级：P0  
> 适用版本：SMS AI Agent V3（ReAct）

---

## 一、产品结论

AI Agent V3 已具备小范围使用能力，但在扩大到全校教师前，必须完成以下四项 P0 整改：

1. 所有 Agent Tool 执行统一的成绩访问权限控制。
2. 分数、排名、人数、变化量等关键数字必须由后端确定性生成或校验。
3. 发送给外部 LLM 的学生数据遵循最小必要原则。
4. V3 异常时仅允许对安全白名单问题执行 V1 降级。

四项整改必须作为同一个安全版本完成验收。任一 P0 未通过，不得扩大使用范围。

---

## 二、背景与当前问题

当前主链路为：

```text
React AI Sidebar
→ Django Agent API
→ MiniMax M3 ReAct 循环
→ 自定义成绩 Tool
→ Django ORM 与确定性计算
→ LLM 组织回答
→ React 展示结果
```

现有系统已经具备 `ScoreAccessService`，普通成绩接口按用户角色限制可见范围；但 Agent API 目前只验证用户是否登录，Agent Tool 尚未统一接入用户权限范围。

同时，V3 当前由 LLM 根据 Tool 结果生成自然语言和 Markdown 表格，且在 LLM 异常时自动回退 V1 规则引擎。这两项行为分别存在关键数字被改写和复杂问题被错误理解的风险。

---

## 三、整改目标

| 目标 | 成功定义 |
|---|---|
| 防止越权 | 用户通过任何问法、追问或 Tool 组合都不能获取权限范围外的数据 |
| 保证数字可信 | 用户看到的关键数字可以逐项追溯到确定性 Tool 结果 |
| 减少外部数据暴露 | LLM 只接收完成当前问题所需的最小数据 |
| 防止错误降级 | V3 不可用时，系统宁可明确失败，也不能用 V1 猜测复杂意图 |
| 支持审计 | 每次查询可以定位用户、权限范围、Tool 链路、降级原因和结果状态 |

---

## 四、范围

### 4.1 本次必须完成

- Agent API 的角色准入与权限上下文构建。
- 现有 8 个 Tool 的统一权限过滤。
- Tool 参数及返回结果的安全校验。
- 关键数字和结构化表格的确定性输出或一致性校验。
- LLM 输入数据最小化与敏感字段移除。
- V1 降级白名单及禁止降级规则。
- 脱敏审计日志。
- 权限、错数、脱敏、降级四类自动化测试。
- 小范围灰度和紧急关闭开关。

### 4.2 本次不做

- 不引入 LangChain 或更换 ReAct 框架。
- 不重做 React Sidebar 和对话管理。
- 不新增学生、家长角色。
- 不新增预测、评价、学情诊断等业务能力。
- 不建设独立 AI 治理平台。
- 不在本次实现跨设备对话同步。

## 五、权限规则

### 5.1 角色权限矩阵

沿用现有 `ScoreAccessService` 权限口径：

| 用户角色 | Agent 可查询范围 | 权限配置缺失时 |
|---|---|---|
| `admin` | 全校 | 不适用 |
| `staff` | 全校 | 不适用 |
| `grade_manager` | `managed_grade` 对应年级 | 拒绝查询 |
| `subject_teacher` | `teaching_classes` 对应班级 | 拒绝查询 |
| 未知角色 | 无 | 拒绝访问 Agent |
| 未登录用户 | 无 | 返回 401 |

### 5.2 权限执行规则

1. 权限判断必须在后端执行，不能依赖前端隐藏入口。
2. 当前登录用户必须贯穿 Agent、Tool 注册表和实际查询函数。
3. 每个 Tool 查询学生、考试、班级或成绩时，必须先应用权限作用域。
4. 后续新增 Tool 默认无权限；只有接入统一安全执行层后才能注册给 LLM。
5. LLM 传入的年级、班级、学生 ID、考试 ID 只代表查询意图，不能作为授权依据。
6. 多轮上下文、追问 payload 和旧对话中的对象都必须重新执行权限校验。
7. V1 降级路径必须执行相同权限规则，不得绕过。

### 5.3 越权反馈

当目标存在但用户无权查看时，统一返回：

> 未找到符合条件且你有权限查看的数据，请检查查询范围或联系管理员。

回复不得暴露以下信息：

- 目标学生或考试是否真实存在。
- 目标所属班级、年级或业务分组。
- 被拒绝的具体权限关系。

## 六、关键数字确定性

### 6.1 关键数字定义

以下内容属于关键数字：

- 分数、总分、均分、最高分、最低分。
- 名次、总人数、并列关系。
- 分差、占比、权重、加权分。
- 分数变化量、排名变化量。
- 样本数量、考试数量和有效记录数量。

### 6.2 产品规则

1. 关键数字必须来自确定性 Tool 的结构化结果。
2. 表格数据应由后端直接构造，不能以 LLM 生成的 Markdown 表格作为唯一数据源。
3. LLM 可以生成结论性文字，但不能新增 Tool 结果中不存在的关键数字。
4. 返回前必须执行数字一致性校验；无法通过校验时不得展示可疑回答。
5. LLM 文字与结构化表格冲突时，以后端结构化结果为准，并重新生成或使用固定文案。
6. 每个结果应保留可追溯的考试、科目、范围和计算口径。

### 6.3 校验失败反馈

> 数据已经完成计算，但结果说明未能通过一致性校验。本次未展示结果，请重新生成。

系统同时记录 `request_id`，不得将未经校验的内容展示给用户。

---

## 七、外部模型数据最小化

### 7.1 禁止发送字段

除非后续经过单独评审，以下字段不得发送给外部 LLM：

- 学号。
- 身份证号、手机号、邮箱、家庭信息。
- 登录账号及权限配置详情。
- 与当前问题无关的学生名单和成绩明细。
- 数据库内部字段、原始 SQL、系统密钥和完整异常堆栈。

### 7.2 最小必要规则

| 场景 | 可发送给 LLM 的数据 |
|---|---|
| 单学生排名 | 必要显示名、考试名、科目、分数、排名、总人数 |
| Top N | 前 N 名必要显示字段，不发送完整榜单 |
| 趋势 | 指定学生和指定考试范围内的必要趋势点 |
| 加权排名 | 两场考试的必要分数、权重和后端计算结果 |
| 群体对比 | 后端聚合指标，不发送两个群体的完整学生成绩 |
| 权限拒绝 | 不发送目标数据，只返回统一拒绝状态 |

### 7.3 本地存储和日志

1. 浏览器持久化内容不得包含原始批量 Tool 返回结果。
2. 审计日志记录参数摘要和结果状态，不记录完整成绩明细。
3. 日志中的姓名按现有运维需要保留或脱敏，但不得记录学号和外部模型密钥。
4. 退出登录时应清除当前浏览器中的 Agent 对话和上下文。

---

## 八、受控降级

### 8.1 允许降级的白名单

只有同时满足以下条件时，才允许 V3 回退 V1：

- 用户权限上下文有效。
- 问题为单轮、明确、简单查询。
- 查询对象、考试、范围和指标均可唯一确定。
- V1 能执行同等权限过滤。
- 不依赖上一轮上下文或追问 payload。

第一版白名单仅建议包含：

- 明确考试、明确范围的 Top N 排名。
- 明确学生、明确考试、明确排名范围的单学生排名。

### 8.2 禁止降级的场景

- 多轮追问或上下文承接。
- 学生趋势、加权排名和群体对比。
- 模糊考试、模糊学生或模糊范围。
- 解释上一轮结果、导出请求、取消和重置。
- 权限校验失败、越权或权限配置缺失。
- Tool 已执行但结果一致性校验失败。
- 无法确认 V1 是否会保持原始查询语义。

### 8.3 禁止降级时的反馈

> AI 成绩分析服务暂时不可用。为了避免返回不准确的结果，本次没有执行降级查询，请稍后重试。

回复应提供“重新生成”，不得伪装成业务上无数据。

## 九、安全执行流程要求

推荐由研发实现统一安全执行层，具体类名和代码组织由技术方案决定：

```text
身份认证
→ 构建用户权限上下文
→ LLM 选择 Tool
→ 校验 Tool 是否允许执行
→ 校验并标准化参数
→ 应用 ScoreAccessService
→ 执行确定性查询与计算
→ 最小化 Tool 返回结果
→ LLM 生成非权威文字说明
→ 后端生成/校验关键数字与表格
→ 记录脱敏审计日志
→ 返回前端
```

任何 Tool 都不能跳过“权限过滤”和“返回结果最小化”两个步骤。

---

## 十、审计与监控

每次请求至少记录：

- `request_id`、用户 ID、角色。
- 本次生效的权限范围摘要。
- 用户问题的脱敏摘要。
- Tool 名称、调用顺序、参数摘要和耗时。
- 是否触发权限拒绝、数据最小化、数字校验。
- 是否降级、降级原因和降级白名单类型。
- 最终状态：成功、追问、拒绝、超时、校验失败、系统错误。

不得记录：

- MiniMax API Key 或其他密钥。
- 完整原始成绩明细。
- 身份证、手机号等敏感字段。
- LLM 隐式推理过程。

建议监控指标：越权拦截次数、数字校验失败率、V1 降级率、平均 Tool 次数、平均响应时间和外部 LLM 错误率。

## 十一、验收标准

### 11.1 权限验收

| 编号 | 场景 | 预期 |
|---|---|---|
| AUTH-01 | `admin` 查询任意年级 | 在数据存在时正常返回 |
| AUTH-02 | `staff` 查询任意年级 | 在数据存在时正常返回 |
| AUTH-03 | 级长查询负责年级 | 正常返回 |
| AUTH-04 | 级长查询其他年级 | 统一返回无权限可见数据，不暴露目标是否存在 |
| AUTH-05 | 科任教师查询任教班级 | 正常返回 |
| AUTH-06 | 科任教师查询非任教班级 | 统一拒绝，不返回学生、考试或成绩信息 |
| AUTH-07 | 通过学生姓名绕过班级限制 | 无法绕过 |
| AUTH-08 | 通过考试 ID 或学生 ID 绕过限制 | 无法绕过 |
| AUTH-09 | 旧对话切换到无权限账号 | 重新校验并拒绝，不沿用旧结果 |
| AUTH-10 | 未登录调用 Agent API | 返回 401 |

### 11.2 数字验收

| 编号 | 场景 | 预期 |
|---|---|---|
| NUM-01 | 单学生排名 | 回复中的分数、排名、总人数与 Tool 结果一致 |
| NUM-02 | Top N | 表格由后端结构化数据生成，顺序和并列规则正确 |
| NUM-03 | 趋势 | 每个趋势点及变化量均可核验 |
| NUM-04 | 加权排名 | 原始分、权重、加权分和名次一致 |
| NUM-05 | 群体对比 | 均分差、占比和群体内排名一致 |
| NUM-06 | 模拟 LLM 修改数字 | 返回前被拦截，不展示错误数字 |
| NUM-07 | LLM 返回额外数字 | 被移除、重新生成或整体拒绝 |

### 11.3 数据最小化验收

| 编号 | 场景 | 预期 |
|---|---|---|
| DATA-01 | 单学生查询 | 外部请求中无学号、联系方式和无关学生数据 |
| DATA-02 | Top 3 | 不向 LLM 发送完整年级榜单 |
| DATA-03 | 群体对比 | 仅发送聚合结果，不发送完整明细 |
| DATA-04 | 越权请求 | 不向 LLM 发送目标学生或成绩数据 |
| DATA-05 | 查看日志 | 无 API Key、学号和完整成绩明细 |
| DATA-06 | 退出登录 | 本地 Agent 消息和上下文被清除 |

### 11.4 降级验收

| 编号 | 场景 | 预期 |
|---|---|---|
| FALLBACK-01 | 明确单学生排名且 V3 超时 | 通过白名单和权限校验后可降级 |
| FALLBACK-02 | 趋势查询且 V3 超时 | 不降级，返回准确性保护提示 |
| FALLBACK-03 | 多轮追问且 V3 超时 | 不降级 |
| FALLBACK-04 | 越权查询且 V3 超时 | 不降级，不能绕过权限 |
| FALLBACK-05 | 数字一致性校验失败 | 不降级，不展示结果 |
| FALLBACK-06 | 降级成功 | 日志中记录降级类型、原因和权限范围 |

## 十二、发布与灰度

### 12.1 发布前置条件

- 四类 P0 自动化测试全部通过。
- 对现有排名、趋势、加权、对比和追问功能完成回归。
- 使用真实角色账号完成一次人工越权测试。
- 确认外部 LLM 请求体不包含禁止字段。
- 全局开关可以立即关闭 V3 Agent。

### 12.2 灰度顺序

1. 产品、研发、测试账号。
2. 3 至 5 名教务人员，观察 5 个工作日。
3. 少量级长、班主任或科任教师，观察 5 个工作日。
4. 产品部重新验收后，再决定是否向全校教师开放。

### 12.3 No-Go 红线

出现以下任意情况，不得发布或扩大灰度：

- 任意 Tool 可以绕过用户权限范围。
- 越权响应暴露目标学生或数据确实存在。
- 用户可见关键数字不在确定性 Tool 结果中。
- 外部 LLM 请求包含学号、联系方式或无关批量成绩。
- 复杂、多轮或越权问题触发 V1 降级。
- 无法通过 `request_id` 定位一次权限拒绝或错误降级。

## 十三、给研发部的交接说明

请研发部基于本 Brief 输出技术方案，至少回答：

1. 当前用户如何贯穿 ReAct 循环、Tool 注册表和底层查询函数。
2. 如何复用 `ScoreAccessService`，避免 8 个 Tool 重复实现权限逻辑。
3. 后端如何生成结构化表格并校验 LLM 文字中的关键数字。
4. 各类 Tool 返回给 LLM 的最小字段集合。
5. V1 降级白名单如何识别，以及如何保证降级路径同样受权限控制。
6. 审计日志保存哪些字段、保留多久、如何脱敏。
7. 如何通过自动化测试证明新增 Tool 默认不能绕过安全执行层。
8. 本次改造涉及的文件、迁移、配置、依赖和发布回滚方案。

产品部评审技术方案后，研发方可进入实施。

## 十四、给测试部的交接说明

测试重点不是验证“正常账号能查到”，而是证明系统在以下情况下仍然不会泄露或错答：

- 用户使用模糊表达、别名、追问和历史上下文尝试越权。
- 用户直接提供学生 ID、考试 ID 或追问 payload。
- LLM 超时、返回非法 Tool 参数、修改数字或重复调用。
- V3 与 V1 降级切换。
- 用户切换账号、切换对话、退出登录和清除缓存。
- 新增 Tool 未主动编写权限代码。

最终测试报告必须分别给出权限、数字、脱敏、降级四项结论。
