# 前端重修方案

> 目标：将所有页面的体验提升到与仪表盘相同的水准。让科任老师、级长、管理员各有一套"为我而做"的体验，而非同一套界面靠权限遮遮掩掩。

---

## 当前基线

**已经做对的事（不改）：**

- 仪表盘页面按角色感知内容，文案随用户角色变化
- AppShell / Sidebar / AuthContext 的骨架是可用的
- Next.js 15 + React 19 + Tailwind v4 + TypeScript + Vitest 技术栈合理

**当前存在的问题（本次修复范围）：**

| 问题 | 涉及文件 | 严重程度 |
|------|----------|----------|
| 成绩管理页是 1178 行上帝组件 | `src/app/scores/page.tsx` | 严重 |
| 侧边栏有 6 条指向不存在页面的死链接 | `src/components/Sidebar.tsx` | 中等 |
| 四种样式方案混合使用 | 全局 | 中等 |
| `page_backup_20260425.tsx` 提交到了仓库 | `src/app/` | 低 |
| 考试页 / 学生页没有角色感知 | `src/app/exams/` `src/app/students/` | 中等 |
| 没有统一 API 客户端 | 全局 | 中等 |
| 科任老师能看到他无权操作的功能入口 | `src/components/Sidebar.tsx` | 中等 |

---

## 1. 三套侧边栏

> 最高优先级。只改一个文件（`src/components/Sidebar.tsx`），三种角色的体验立刻完全不同。今天改明天见效。

### 现状

当前 Sidebar 是同一套菜单，只是个别按钮通过 `canWriteScores(user)` 控制显隐。三种角色看到的菜单结构完全一样：

```
所有角色看到的侧边栏（当前）：
  Dashboard
  教学管理
    学生信息
    考试管理    ← 科任老师也能看到（虽然他不能新建考试）
    成绩管理    ← 科任老师也能看到（虽然只能看，不能管理）
    成绩查询
    成绩分析
      班级/年级分析
      个人成绩分析
    目标生筛选
      简单筛选
      高级筛选
      我的规则
      变化追踪
  系统管理
    系统设置    ← 死链接
    报表管理    ← 死链接
    数据备份    ← 死链接
  其他功能
    帮助中心    ← 死链接
    技术支持    ← 死链接
    毕业管理    ← 死链接
```

### 修改后

三种角色各一套菜单：

**管理员：** 看到所有管理功能。他管全校。

```
  Dashboard — 全校工作台
  学生管理      ← 全校学生的增删改查
  考试管理      ← 新建/编辑/删除考试
  成绩管理      ← 录入/导入/编辑/删除成绩
    成绩查询    ← 查全校成绩
    成绩分析    ← 全校分析
  目标生筛选
  校历事件
```

**级长：** 管理自己年级。入口命名体现"年级"视角。

```
  Dashboard — 九年级工作台
  年级学生      ← 只看自己年级的学生（可管理）
  年级考试      ← 创建/编辑自己年级的考试
  年级成绩      ← 录入/导入/编辑自己年级的成绩
  成绩查询      ← 查自己年级成绩
  年级分析      ← 对比各班、追踪年级趋势
  目标生筛选    ← 找出年级里需要关注的学生
```

**科任老师：** 只能查和分析，不能管。入口命名体现"任教"视角。

```
  Dashboard — 工作台
  任教学生      ← 只看任教班级的学生（只读）
  成绩查询      ← 查任教班级的成绩
  成绩分析      ← 分析任教班级的表现
    班级对比
    个人追踪
  目标生筛选    ← 找出任教班级里需要关注的学生
```

## 2. 成绩管理页改成工作台

> **使用范围**：仅面向管理员和级长。科任老师不应看到此页面（侧边栏里不出现"成绩管理"入口）。

### 现状

`scores/page.tsx`（路径 `/scores`）是一个 1178 行的单文件组件，职能是管理成绩——手动录入、Excel 批量导入、编辑、批量删除。这是管理员和级长的操作工具。

打开后默认显示空表格 + "目前没有任何成绩记录"，上方堆着年级/考试/班级/学号/姓名五个筛选项、全选/删除筛选/删除选中三个批量操作按钮、手动新增/批量导入两个入口按钮。

问题：
- 空状态没有引导——用户不知道第一步该做什么
- 核心任务（录入/导入）被埋在一堆筛选和批量操作中
- 级长默认看到全校数据，需要自己筛到自己年级
- 批量操作栏始终可见，即使还没勾选任何记录

### 修改内容

将成绩管理页从"数据表格+筛选面板"改为**操作工作台**：

| 区域 | 修改前 | 修改后 |
|------|--------|--------|
| 页面标题区 | `成绩管理 + 两个按钮` | 不变，保留 |
| 第一屏内容 | 筛选栏 + 空表格 | **三个操作卡片（录入/导入/查看进度）** |
| 空状态 | `目前没有任何成绩记录` | 三个大卡片引导第一步操作 |
| 搜索入口 | 嵌在筛选栏里 | **独立的快速搜索栏**，放在最显著位置 |
| 筛选区域 | 永远展开 | **折叠式**，点击"高级筛选"展开 |
| 最近动态 | 没有 | **最近导入记录 + 待完成录入列表** |
| 角色默认值 | 无 | **级长默认筛选管理年级** |
| 批量操作 | 始终可见 | **有勾选时才出现**批操作栏 |

## 5. 建 API 层

### 现状

前端与后端通信方式：每个页面组件内直接写 `fetch()`。没有统一的请求管理、token 刷新、错误处理、加载状态。

具体问题：
- Token 以 query string `?token=xxx` 传递（仪表盘）——暴露在 URL 里
- 有的请求用 `Authorization: Bearer xxx` header，有的用 `?token=xxx` ——不一致
- Token 过期后各组件自己处理（或不处理）
- 加载状态靠组件自己的 `useState(isLoading)` 管理 —— 每个组件重复写一遍
- 请求 URL 拼接散落在各处：`http://localhost:8000/api/`、`http://${window.location.hostname}:8000/api/`、`/api/scores/` ——三种写法

### 修改内容

#### 5.1 统一请求客户端

新建 `src/lib/api.ts`：

```typescript
const API_BASE = '/api'; // 依赖 Next.js rewrites 代理到 Django

// 类型
type ApiOptions = {
  params?: Record<string, string | number>;
  body?: unknown;
  headers?: Record<string, string>;
};

type ApiResponse<T> = T;

// 核心方法
const request = async <T>(
  method: string,
  path: string,
  options: ApiOptions = {}
): Promise<ApiResponse<T>> => {
  const token = getToken(); // 从 localStorage 或 context 获取

  // 构建 URL，params 自动追加
  const url = buildURL(API_BASE + path, options.params);

  const res = await fetch(url, {
    method,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...options.headers,
    },
    ...(options.body ? { body: JSON.stringify(options.body) } : {}),
  });

  // 统一错误处理 —— token 过期自动尝试刷新
  if (res.status === 401) {
    const refreshed = await tryRefreshToken();
    if (refreshed) {
      return request(method, path, options); // 重试
    }
    logout();
    throw new Error('认证已过期');
  }

  if (!res.ok) {
    const error = await res.json().catch(() => ({}));
    throw new ApiError(res.status, error.message || '请求失败');
  }

  return res.json();
};

// 封装好的方法
export const api = {
  get: <T>(path: string, params?: Record<string, string | number>) =>
    request<T>('GET', path, { params }),

  post: <T>(path: string, body?: unknown) =>
    request<T>('POST', path, { body }),

  put: <T>(path: string, body?: unknown) =>
    request<T>('PUT', path, { body }),

  delete: <T>(path: string) =>
    request<T>('DELETE', path),

  // 文件上传
  upload: <T>(path: string, formData: FormData) =>
    request<T>('POST', path, {
      body: formData,
      headers: { 'Content-Type': '' }, // 让浏览器自动设置 multipart
    }),
};
```

**关键点：**
- 统一从 `API_BASE` 发请求，走 Next.js 的 `rewrites` 代理 → 没有跨域问题
- Token 自动从 localStorage 读取并附加到 header → 不再用 `?token=` 传
- 401 自动尝试刷新 token → 组件不需要自己处理
- 参数自动序列化到 URL → 不用手动拼 `new URLSearchParams()`
- 统一错误类型 → 可以全局捕获做 toast

#### 5.2 替换现有 fetch 调用

按页面优先级逐个替换：

| 顺序 | 页面/模块 | 替换方法 |
|------|-----------|----------|
| 1 | `AuthContext.tsx` | `fetch('/api/users/me/')` → `api.get('/users/me/')` |
| 2 | `page.tsx`（仪表盘） | 三处 fetch → 三处 api.get |
| 3 | `scores/page.tsx`（改组件时顺便） | 所有 fetch → api |
| 4 | `students/` 下所有页面 | 所有 fetch → api |
| 5 | `exams/` 下所有页面 | 所有 fetch → api |
| 6 | `analysis/` 下所有页面 | 所有 fetch → api |
| 7 | `target-students/` 下所有页面 | 所有 fetch → api |

#### 5.3 清理工作

替换完成后需要处理：
- 删除各处手动拼接的 `backendBaseUrl` 变量
- 删除各处手动构建的 `new URLSearchParams()` 代码
- 统一删除 `?token=${encodeURIComponent(token)}` 的 query string 传参
- 仪表盘页的 `getBackendBaseUrl()` 函数删除
- 所有 `const backendBaseUrl = ...` 声明删除
